# Decisions (ADR) — самообновляемый харнесс

Формат: Контекст / Решение / Последствия. Кратко.

## ADR-1: Гибрид (3 слоя), не монорепа

**Контекст.** Инстансы (abdulpay, fenris, turbo-omni) — отдельные GitLab-репо с разными remote.
Дрейф умеренный (abdulpay≈темплейт, turbo-omni разошёлся, fenris форк).
**Решение.** Не сливать в монорепу. 3 слоя: CORE (синк жёстко) / LANG (по языку) / LOCAL (свобода).
**Последствия.** Синкаем только CORE → нет Copier-конфликтов на локальных правках, работает даже на форке.

## ADR-2: Copier, не cookiecutter / git-subtree

**Контекст.** Нужна раскатка обновлений CORE вниз, не разовый скаффолд.
**Решение.** Copier (`copier update` умеет template→project merge; cookiecutter — нет).
**Последствия.** Зависимость Python/pipx. Хрупкость на конфликтах — митигируется синком только CORE.

## ADR-3: Обратный канал — отдельный процесс, не Copier

**Контекст.** Copier односторонний by design. Нужен подъём улучшений инстанс→темплейт.
**Решение.** `scripts/harness-contribute.sh` копирует CORE-изменения + печатает diff и инструкцию PR.
**Последствия.** Скрипт слеп к направлению (может поднять старую версию) → показывает diff,
git-операции и решение — за человеком. Не автоматизируем PR.

## ADR-4: settings.json — НЕ в CORE-sync (осознанно)

**Контекст.** Структура хуков общая (CORE-природа), но `settings.json` содержит проектные
плейсхолдеры (`<TEST_CMD>`, `<WIKI_PATH>`) и локальные отличия (fenris: свои guard/sensor).
**Решение.** Пока НЕ синкать `settings.json` через Copier. Раскатка — через `.template` при setup.
Якорь путей унифицирован отдельно: все хуки используют `$CLAUDE_PROJECT_DIR` (не `./`, не `${REPO_ROOT}`,
не хардкод) — это исправлено вручную во всех инстансах + темплейте.
**Последствия.** Структура хуков может дрейфовать. Пересмотреть, когда понадобится синк хуков —
тогда нужен аккуратный шаблон с плейсхолдерами, не прямой синк.

## ADR-5: Версия — git-тег темплейта

**Контекст.** Copier версионирует по git-тегам; без тегов пишет `_commit: <hash>` (нечитаемо).
**Решение.** Версия в `VERSION` (0.1.0) + git-тег `v0.1.0` на темплейте для читаемого `copier update`.
**Супердано 23.06:** файл `VERSION` удалён — ручной бамп отставал от тега 4 раза подряд, а copier
читает только теги (VERSION никто не парсил). Единственный источник версии = git-тег `vX.Y.Z`.
**Последствия.** Тег создаёт владелец вручную (git-операции вне зоны агента):
`git -C harness-template tag v0.1.0`.
