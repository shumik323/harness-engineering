# end-session

Обновить лог долгосрочной памяти по итогам сессии.

## Шаг 0 — Triage буфера наблюдений (перед записью лога)

1. Прочитать `.claude/PENDING-NOTES.md` (если есть и непустой).
2. Каждую заметку классифицировать по адресу:
   - **One-off** (разовый факт/баг) → войдёт в `log.md` (Шаг 3)
   - **Recurring rule / gotcha** → новый `§N` в `.claude/docs/gotchas.md`
     (формат — см. шапку файла)
   - **Архитектурное решение** → ADR в `$WIKI_PATH/decisions.md`
3. **Gate generic / instance** (для каждого recurring rule / gotcha — это и есть подъём боли в темплейт):
   спросить «эту боль словит ЛЮБОЙ проект на этом харнессе, или это специфика стека/проекта?»
   - **instance** (специфика стека/проекта) → только `.claude/docs/gotchas.md`. Стоп.
   - **generic + это ПРАВИЛО** (always-do / never-do, не зависит от стека) → записать локально в
     `gotchas.md` И пометить кандидатом на CORE: предложить добавить в `.claude/rules/common/*.md`
     (workflow/testing/git — они синкаются во все инстансы). Подъём в темплейт — `scripts/harness-contribute.sh`
     (печатает diff, git за человеком). Это самоулучшение CORE.
   - **generic + это ПАТТЕРН / quirk** (полезный контекст, не жёсткое правило) → записать локально И
     предложить добавить в `skeleton/.claude/docs/gotchas.md.template` (будущие инстансы получат при скаффолде).
   - Не уверен generic/instance → instance по умолчанию (не засорять CORE сомнительным).
4. Показать раскладку пользователю на ревью (включая generic-кандидатов и куда они метятся) →
   дождаться подтверждения → записать.
5. Очистить `PENDING-NOTES.md` до шапки (заметки разнесены).

## Что сделать

1. Прочитать `$WIKI_PATH/log.md` (путь из `.harness.conf`)
2. Определить что было сделано за сессию (из контекста беседы)
3. Добавить запись в конец `log.md`:

```
## [YYYY-MM-DD] тип | краткое описание
```

Типы: `build` / `fix` / `update` / `query` / `refactor`

Структура записи:
- **Что сделал** — конкретные изменения
- **Что удивило** — нюансы, неожиданное поведение
- **Что завтра** — следующий шаг

4. Если появился новый паттерн или решение → предложить добавить в `learnings.md` или `decisions.md`

## Важно

- Не переписывать предыдущие записи — только append
- Сверить "Что завтра" из предыдущей записи: выполнено / перенесено?
- Если перенесено → добавить раздел **Перенесено:** в начало "Что завтра" новой записи
