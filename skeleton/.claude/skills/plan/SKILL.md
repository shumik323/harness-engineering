# plan

Шаблон плана перед выходом из планирования в код. Заполни все секции, удали комментарии.

**Три секции ниже — ОБЯЗАТЕЛЬНЫЕ.** Пустая обязательная секция = план невалиден → СТОП,
из планирования не выходим. «Допущений нет» / «границ нет» — это явный ответ, не пустота.
Молчаливый пропуск секции — не разрешён.

---

## Шаблон

```markdown
## Context (что уже есть)

<!-- Стек, файлы, текущее поведение. Что важно знать, чтобы не сломать существующее. -->

## Изменения

<!-- Что и где меняется. Файлы, символы, последовательность. -->

## Допущения и развилки   ← ОБЯЗАТЕЛЬНО

<!-- Каждое решение, от которого зависит код. Развилка = выбор, который можно сделать
     по-разному и который меняет реализацию (когда считать пользователя авторизованным;
     откуда брать форму ответа; как обрабатывать пустой список).
     Необъявленная развилка = СТОП на владельца — не зашивай выбор в план молча.
     Нет развилок — напиши явно «Допущений нет». -->

## Внешние контракты   ← ОБЯЗАТЕЛЬНО для boundary-фич

<!-- Для фичи, пересекающей границу (сеть, API, env, чужой сервис):
     request / response / errors / CORS / env-переменные.
     Контракт РЕАЛЬНЫЙ — взят у живого бэка/доки, не stub и не выдуман.
     Фича не трогает границу — напиши «Границ нет». -->

## Teardown / failure   ← ОБЯЗАТЕЛЬНО

<!-- На каждый happy-path — парный путь сбоя. Что чистится при ошибке (токены, сессия,
     подписки, временные файлы). Что показываем юзеру при сбое границы.
     Проектируется здесь, а не затыкается реактивно потом. -->

## Verification

<!-- Чем закрывается план: запускаемая проверка (тест / сборка / линт / ручной прогон).
     Для boundary — в реальной среде (прод-конфиг, реальный .env), не в удобном транзиентном. -->
```

## Почему эти три секции обязательны

Источник — разбор провалов, где план честно исполнял зашитые в него дыры:

- **Допущения и развилки** — молчаливый design-выбор зашивается в план и дальше исполняется
  как факт. Секция делает выбор видимым → его можно вынести на владельца до кода.
- **Внешние контракты** — правило verify стоит на выходе, ничто не требует контракт границы
  на входе. Stub-форма ответа всплывает реактивно. Секция тащит реальный контракт вперёд.
- **Teardown / failure** — happy-path проектируют, cleanup забывают. Failure всплывает трижды
  по ходу. Парность пути сбоя проектируется здесь.

Машина это не форсит (что есть «развилка» — суждение). Форсит ритуал: пустая секция → СТОП.
Риск гейминга («допущений нет» формально) — на дисциплине, не на хуке. Граница:
машина форсит факты, ритуал — видимость суждения, правду — человек.
