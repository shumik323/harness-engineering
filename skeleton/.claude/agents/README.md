# .claude/agents — роли-агенты (нативный Claude Code)

Субагенты Claude Code: markdown + YAML-фронтматтер (`name`, `description`, `tools`, `model`,
опц. `isolation`). Зовутся через Task tool по имени. Один уровень вложенности (агент не
вызывает агента).

## Что здесь лежит

- `bug-triage.md` — **scaffold**, обобщён из боевого fenris-агента. Read-only следователь:
  grep gotchas → locate → (diff с reference) → гипотеза → структурный отчёт. **Не редактирует код.**
  При первом использовании в проекте — заполнить плейсхолдеры, навести KB-секцию на реальные пути.

## Слой и синк

Агенты — **instance-owned scaffold**, не CORE. Каждый проект кастомизирует роль под свой
стек и базу знаний (fenris это доказал: свои пути, learnings.md, VeeValidate-специфика). Поэтому
`.claude/agents` исключён из Copier-синка (`copier.yml` → `_exclude`), как и `.claude/docs`:
обновления метода НЕ форсятся сверху.

**Доставка — reference-by-copy, не авто.** Раз `agents/` исключён из copier, `copier copy/update`
его в инстанс НЕ кладёт. Это эталон: когда инстансу нужна роль — скопировать файл руками в свой
`.claude/agents/`, заполнить плейсхолдеры, дальше владеть им сам. Авто-доставки при init нет
(в отличие от CORE) — это осознанно: пустой агент-плейсхолдер в каждом репо вреднее, чем явное
принятие роли по нужде.

## Как завести новую роль

1. Скопировать `bug-triage.md` как образец структуры (фронтматтер + метод + guardrails + output).
2. Сузить `tools` до минимума роли. Read-only роль → без `Write`/`Edit`; следящая роль → без
   side-effect Bash (`npm run`, тест-раннеры).
3. Прописать guardrails явно: что агент НЕ делает. Read-only роль обязана останавливаться на
   описании правки, не на её внесении.
4. Output format — жёсткий шаблон, чтобы оркестратор парсил ответ предсказуемо.

## Сверено с доками Claude Code

Источник: https://code.claude.com/docs/en/sub-agents.md. Что важно знать, чтобы не дорабатывать:

- **`tools` — это enforcement, инструкция в промпте — нет.** Отсутствие `Write`/`Edit` в allowlist
  реально блокирует правки. А вот «Bash только для git» промптом НЕ форсится: если `Bash` в списке,
  агент может запустить что угодно. Жёсткая гарантия = убрать `Bash` или `permissionMode: auto`
  (classifier вет ит команды).
- **`description` = триггер автоделегации**, не «что делает». `bug-triage` намеренно БЕЗ
  «Use proactively» — триаж зовут явно, а не на каждое упоминание ошибки. Хочешь автозапуск роли —
  добавь «Use proactively when…».
- **`model: inherit`** по умолчанию (если поле опущено). Явный `sonnet`/`haiku`/`opus` —
  осознанный выбор стоимости/мощности.
- **`memory: project`** — опция для роли, которая копит знание между сессиями (например, триаж копит
  классы багов). Пишет в `.claude/agent-memory/<name>/`, шарится через git. Не включена в scaffold —
  добавь, если роль должна учиться.
- **Загрузка при старте сессии.** Правка `.md` напрямую не подхватывается на лету — рестарт сессии
  или интерфейс `/agents` (применяет сразу).
- **Plugin-агенты теряют** `hooks`/`mcpServers`/`permissionMode`. Эти агенты — проектные
  (`.claude/agents/`), не plugin, поэтому поля работают.

## Чего здесь намеренно НЕТ

Роли `verifier` / `bugfixer` / оркестратор — **по реальной боли, не впрок** (`rules/common/workflow.md`:
build from failures). Под них пока нет боевого артефакта, который стоит обобщать. `bug-triage` есть —
поэтому он здесь. Появится повторяющаяся нужда в проверяющем/фиксящем агенте — заводить по этому же
образцу.
